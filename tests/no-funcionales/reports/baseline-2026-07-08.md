# Resultado de Referencia (Baseline) — Prueba de Carga, 6 Módulos

**Fecha**: 2026-07-08
**Script**: `k6/seis-modulos-carga.js`
**Ambiente**: XAMPP local (Apache + MySQL), `http://localhost/QA-OpenCart-Testing/upload`
**Base de datos**: `opencart_upload` (instalación limpia vía `install/cli_install.php`)
**Duración**: ~3m 30s

## Resumen

Los 10 umbrales definidos se cumplieron para los 6 módulos funcionales del proyecto. Sin
errores HTTP ni errores de negocio, con hasta 59 usuarios virtuales concurrentes combinados
entre todos los escenarios.

> Nota: reemplaza el baseline anterior de esta misma fecha, que corrió por error contra
> `opencart/` — esa copia del código tiene plantillas `.html` en vez de `.twig` y nunca llegó a
> renderizar HTML real (devolvía un error de Twig con status 200). Ver `README.md` sección 0
> para el detalle del hallazgo y cómo se corrigió instalando `upload/`.

## Notas del ambiente

- Esta corrida requirió instalar OpenCart en `upload/` desde cero (config.php, base de datos
  `opencart_upload`, `composer install` para Twig). Ver sección 0 del `README.md` de esta
  carpeta para los pasos completos.
- La contraseña del usuario admin (`user_id=1`) se fijó a `admin123` en la instalación; a
  diferencia de `opencart/`, no se observó que cambiara espontáneamente en `upload/` durante
  esta sesión.
- La corrida crea registros reales: cuentas de cliente nuevas (módulo Login y Registro) y
  reseñas nuevas pendientes de aprobación (módulo Reseñas) en la base de datos
  `opencart_upload`. Si se repite muchas veces, considerar limpiar `oc_customer` / `oc_review`
  o reinstalar la base de datos.
- El máximo aislado de 804ms en `duracion_catalogo_busqueda` no afectó el p95 ni el umbral,
  pero conviene repetir la corrida para confirmar si es un outlier o un patrón bajo cierta
  concurrencia específica de búsqueda.

## Conclusión

Los 6 módulos funcionales del proyecto (Login/Registro, Catálogo/Búsqueda, Carrito, Checkout,
Gestión de Inventario, Reseñas) cumplen los criterios de rendimiento definidos para carga
concurrente moderada (hasta 59 usuarios virtuales combinados). No se identificaron cuellos de
botella que requieran una acción correctiva inmediata. Se recomienda repetir esta prueba tras
cambios significativos en el código de cada módulo, o antes de un despliegue a producción con
volúmenes reales estimados.
